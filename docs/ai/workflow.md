# Claude Code Workflow

이 문서는 Claude Code를 사용해 팀 프로젝트를 진행할 때 따르는 공통 작업 흐름을 정의한다.

---

## 1. 작업 전 플로우

### 1.1 루트 디렉토리에서 실행

Claude Code는 프로젝트 루트 디렉토리에서 실행한다.

```bash
cd <project-root>
claude
```

---

### 1.2 공통 규칙 확인

작업 시작 전 다음 문서를 확인한다.

- `CLAUDE.md`
- `docs/architecture.md`
- `docs/conventions.md`
- `docs/api-spec.md`
- `docs/db-schema.md`

작업 성격에 따라 필요한 문서를 우선적으로 읽는다.

| 작업 유형 | 우선 확인 문서 |
|---|---|
| API 추가/변경 | `docs/api-spec.md` |
| DB 변경 | `docs/db-schema.md` |
| 구조 변경 | `docs/architecture.md` |
| 코드 작성 | `docs/conventions.md` |
| 테스트 작성 | `docs/conventions.md` |
| Claude Code 작업 방식 확인 | `docs/ai/workflow.md` |

---

### 1.3 작업 범위 정의

작업 시작 전 아래 항목을 명확히 한다.

- 작업 목표
- 담당 영역
- 수정 가능 파일
- 수정 금지 파일
- 예상 영향 범위
- 완료 기준
- 테스트 방법

예시:

```text
[작업 목표]
대화 생성 API 구현

[담당 영역]
Backend

[수정 가능 파일]
backend/src/main/...
backend/src/test/...
docs/api-spec.md

[수정 금지 파일]
frontend/
rag-pipeline/

[완료 기준]
POST /api/conversations API가 동작하고, Service 테스트와 Controller 테스트가 통과한다.
```

## 2. Plan Mode 플로우

### 2.1 구현 전 Plan 작성

구현 전에 Claude Code에게 먼저 Plan을 작성하게 한다.

Plan에는 다음 내용이 포함되어야 한다.

- 작업 요약
- 변경 대상 파일
- 구현 단계
- 테스트 계획
- 문서 수정 필요 여부
- 위험 요소
- 완료 기준

예시 프롬프트:

```text
이 기능을 구현하기 전에 먼저 Plan을 작성해.
수정할 파일, 구현 단계, 테스트 계획, 문서 수정 필요 여부를 나눠서 정리해.
아직 코드는 수정하지 마.
```

---

### 2.2 Plan 검토 후 구현

Plan을 확인한 뒤 구현을 진행한다.

작업 범위가 크면 한 번에 구현하지 않고 milestone 단위로 나눈다.

예시:

```text
Plan의 1단계만 먼저 구현해.
구현 후 관련 테스트를 실행하고 결과를 알려줘.
```

---

## 3. 구현 플로우

### 3.1 Acceptance Criteria 정리

구현 전 Acceptance Criteria와 테스트 케이스를 먼저 정리한다.

```md
## Acceptance Criteria

- 사용자는 질문을 입력할 수 있다.
- 서버는 질문과 대화 ID를 저장한다.
- RAG Pipeline 호출 결과를 응답으로 반환한다.
- 실패 시 공통 Error Response를 반환한다.

## Test Cases

- 정상 요청 시 대화 메시지가 저장된다.
- 필수 값이 없으면 400 응답을 반환한다.
- RAG Pipeline 호출 실패 시 502 응답을 반환한다.
```

---

### 3.2 실패 테스트 우선 작성

기능과 아키텍처가 명확한 경우, 핵심 테스트를 먼저 작성한다.

우선순위는 다음과 같다.

1. 도메인 로직 테스트
2. Service 테스트
3. API Contract 테스트
4. Integration 테스트
5. E2E 테스트

---

### 3.3 최소 구현

테스트를 통과시키기 위한 최소 구현을 먼저 한다.

- 불필요한 추상화 금지
- 과도한 공통화 금지
- 사용되지 않는 코드 추가 금지
- 미래 확장성을 이유로 현재 필요 없는 구조 추가 금지

---

### 3.4 리팩토링

테스트가 통과한 뒤 필요한 경우 리팩토링한다.

리팩토링 시 다음을 지킨다.

- 동작 변경과 구조 변경을 한 커밋에 섞지 않는다.
- 테스트가 깨지지 않는 범위에서 진행한다.
- 기존 컨벤션과 아키텍처를 따른다.

---

## 4. 검증 플로우

작업 후 아래 순서로 검증한다.

```bash
./scripts/format.sh
./scripts/lint.sh
./scripts/test.sh
./scripts/verify.sh
```

각 명령의 역할은 다음과 같다.

| 명령 | 목적 |
|---|---|
| `format.sh` | 코드 스타일 자동 정리 |
| `lint.sh` | 코드 품질 및 규칙 위반 검사 |
| `test.sh` | 테스트 실행 |
| `verify.sh` | 전체 검증 통합 실행 |

---

## 5. 작업 완료 플로우

작업 완료 전 다음을 확인한다.

- `git diff`로 변경 범위 확인
- 요청 범위 외 변경 여부 확인
- 테스트 통과 여부 확인
- 문서 수정 필요 여부 확인
- 불필요한 로그, 임시 코드 제거
- 작업 결과 요약 작성

작업 결과 요약에는 다음을 포함한다.

```md
## 작업 결과

### 변경 사항
-

### 수정 파일
-

### 실행한 명령
-

### 테스트 결과
-

### 문서 수정
-

### 남은 TODO
-
```

## 6. 세션 운영 원칙

### 6.1 1 change-set = 1 session

하나의 세션에서는 하나의 변경 단위만 다룬다.

좋은 예:

- 로그인 API 구현
- 대화 저장 Service 구현
- RAG Pipeline 호출 Client 구현
- 프론트 채팅 입력 컴포넌트 구현

나쁜 예:

- 로그인, 채팅, 관리자 페이지, DB 구조를 한 번에 수정
- 기능 구현과 대규모 리팩토링을 동시에 진행

---

### 6.2 새 세션 시작 기준

다음 경우 새 세션을 시작한다.

- 하나의 기능이 완료된 경우
- 작업 범위가 다른 모듈로 넘어가는 경우
- 대화가 길어져 맥락이 복잡해진 경우
- 기존 작업과 별개의 버그를 수정하는 경우

---

### 6.3 세션 간 인수인계

새 세션을 열기 전 다음 정보를 남긴다.

- 완료한 작업
- 수정한 파일
- 실행한 테스트
- 실패한 테스트와 원인
- 다음 작업에서 이어갈 내용

필요 시 `docs/ai/current-plan.md`와 `docs/ai/working-log.md`에 기록한다.

---

## 7. 금지 사항

- 테스트 실패를 무시하고 완료 처리하지 않는다.
- 실패한 테스트를 삭제해서 통과시키지 않는다.
- 문서와 다른 API를 구현하지 않는다.
- DB Schema 변경 후 문서를 수정하지 않은 상태로 끝내지 않는다.
- 다른 팀원 담당 영역을 임의로 수정하지 않는다.
- Secret, Token, Credential을 코드에 포함하지 않는다.
- Claude Code가 제안한 내용을 검토 없이 그대로 반영하지 않는다.
